# Agentic RAG Evaluator

A fully local agentic RAG system with a 5-layer evaluator. No API keys, no cloud — everything runs on your machine via [Ollama](https://ollama.com).

## Stack

| Component | Technology |
|-----------|-----------|
| LLM (generator + judge) | `llama3.2` via ChatOllama |
| Embeddings | `nomic-embed-text` via OllamaEmbeddings |
| Vector DB | ChromaDB (persistent local storage) |
| Agent orchestration | LangGraph |
| Evaluation | Custom 5-layer scorer with LLM-as-judge |

## Architecture

```
Query
  │
  ▼
memory_check ──► planner
                    │
          ┌─────────┼─────────┐
          ▼         ▼         ▼
   vector_retriever  file_retriever  api_retriever
   (ChromaDB)        (notes/*.txt)   (api_mock.json)
          │         │         │
          └────►  evaluator ◄─┘
                    │
            ┌───────┴───────┐
            │               │
         refine?          proceed
            │               │
         planner         generator
                            │
                       memory_update
                            │
                           END
```

**Parallel fan-out**: when the planner selects multiple sources, the retriever nodes run concurrently. LangGraph's `Annotated` reducers safely merge their results back into a single state.

## Data Sources

- **`data/docs/`** — 5 AI/ML markdown knowledge base files (RAG, chunking, embeddings, vector DBs, LLM evaluation)
- **`data/notes/`** — 3 internal experiment notes (model comparison, chunking experiments, evaluation notes)
- **`data/api_mock.json`** — mock "latest AI" endpoint with recent models, benchmarks, and trending techniques

## 5-Layer Evaluator

| Layer | Name | Weight | What it measures |
|-------|------|--------|-----------------|
| L1 | Plan Quality | 0.20 | Did the agent select the right sources? (Jaccard + LLM judge) |
| L2 | Retrieval Quality | 0.25 | Were the retrieved chunks relevant? (LLM judge per source) |
| L3 | Refinement Quality | 0.20 | Did re-retrieval improve relevance? |
| L4 | Memory Efficiency | 0.15 | Was the memory cache used correctly? |
| L5 | Answer Alignment | 0.20 | Is the answer faithful and relevant? (0.6×faithfulness + 0.4×relevance) |

**Overall score** = weighted average of all 5 layers.

## Setup

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Pull Ollama models

```bash
ollama pull llama3.2
ollama pull nomic-embed-text
```

### 3. Index documents and generate test set

```bash
python3 ingest.py
```

This will:
- Chunk all docs (500 tokens, 50 overlap) and embed them into ChromaDB
- Auto-generate `data/test_set.json` (up to 20 test cases) using the LLM

## Usage

### Run a single query

```bash
python3 run_agent.py "What are the best chunking strategies for RAG?"
```

Output includes: plan chosen, sources hit, iterations taken, final answer, and full trace.

### Run the full evaluation

```bash
python3 run_eval.py
```

Output includes:
- Per-query scores across all 5 layers
- Rich table with per-layer averages
- Weak spot detection (flags any layer average below 0.70)
- Full trace for the worst-performing query
- `agentic_eval_report.json` saved to the project root

## Key Design Notes

- **No cloud, no API keys** — everything runs locally via Ollama
- **Graceful failures** — all Ollama calls are wrapped in `try/except`; defaults are used on failure
- **Memory caching** — successful plans (avg relevance ≥ 0.75) are cached; reused when word overlap > 60%
- **Refinement loop** — if retrieval scores are below the confidence threshold, the agent re-plans (up to `max_iterations`)
- **ChromaDB persistence** — embeddings stored in `chroma_db/` and reused across runs
